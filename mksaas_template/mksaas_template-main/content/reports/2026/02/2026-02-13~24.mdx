## 2月13日：OpenClaw 落地避坑、AI 编程模型"降本增效"与账号风控破解

本日群聊重点围绕 **AI Agent（数字员工）的实战配置** 展开。大家深入讨论了 OpenClaw 的高昂 Token 成本控制方案，对比了 Opus 4.6、MiniMax 2.5 等模型在编程场景下的性价比，并针对 Claude Code 封号问题贡献了一套实用的"救火"方案。

### 1. 技术讨论：AI 编程模型的选择与"降本"策略

关于编程和办公自动化场景下的模型选择，群内形成了一套详细的分层使用共识，旨在平衡效果与成本：

* **主力模型与"备胎"策略：**

  * **Coding 场景：** 国产模型 MiniMax-M2.5 和 Qwen3-Coder 被视为"黑马"，编程能力强且成本低（约 $0.3/M），可作为主力模型；而昂贵的 Codex / Claude Code 则建议作为 Fallback（兜底）选项，仅在大代码库或复杂逻辑时调用。

  * **办公自动化：** Kimi K2.5 备受推崇，群友反馈其在浏览器操作和多线程自动化上表现稳定，且价格仅为 Opus 的 1/8。

  * **心跳/巡检任务：** 严禁使用昂贵模型。建议使用 Gemini Flash（免费额度高）或 DeepSeek，避免一晚上烧掉几十美金。

* **Opus 4.6 的配置陷阱：**

  * 有群友反馈使用 Opus 4.6 时出现 `prompt too long` 提示且无法自动压缩。

  * **解决方案：** 这通常意味着模型缓存未开启或反代接口不支持。需要检查 Max Token 配置，或尝试手动发送 `/compact` 命令进行压缩。

### 2. 实战避坑：OpenClaw（数字员工）的配置与成本

多位群友分享了部署 OpenClaw 的实战经验，核心痛点集中在"烧钱"和配置上：

* **Token 消耗预警：** OpenClaw 极其消耗 Token，有群友测试 Kimi 2.5 基础套餐半天即被限流。

  * **避坑建议：** 尽量不要直接用开放平台的 API（较贵），推荐使用 Kimi Coding 类的专用套餐，或者购买包月服务以应对高频调用。

* **实际应用效果：** 尽管配置繁琐，但在信息检索方面表现出色。有案例显示，只需发送链接，数字员工即可自动登录星球并检索指定帖子内容，无需人工干预。

### 3. 风控对策：Claude Code 账号被封怎么办？

针对近期 Claude Code 频繁封禁谷歌账号的问题，群内总结出了一套可行的注册方案：

* **问题背景：** 许多用户的谷歌账号被封，且 Claude 强制要求谷歌账号登录。

* **破解方案：** 谷歌账号**不一定**非要是 Gmail 结尾。

  * 建议注册一个 Outlook 邮箱，然后将其作为第三方邮箱申请成为谷歌账号。

  * 实测使用这种"Outlook 注册的谷歌号"登录 Claude Code，存活率较高，且即便被封也不影响主 Gmail 账号的安全。

## 2月14日~24日：语音输入法实战比拼、支付风控预警与 Claude Code 降本增效

本期群聊技术浓度与实战经验极高。大家从昂贵的"语音输入法"平替方案聊到了支付渠道的残酷风控，再到 Claude Code (CC) 的 token 优化与防封号技巧。

### 1. 工具测评：语音输入法的"三国杀"

针对高频使用的语音转文字（口喷代码/写作），群友们对市面上的主流工具进行了深度横评：

* **Typeless（高端付费流）：**

  * **优点：** 智能程度高，支持结构化输出（自动去除语气词），自带强大的词库（如能识别 Claude.md），上下文理解能力强，无需频繁手动修正。

  * **缺点：** 价格较贵（约 $20/月），手机端体验一般（缺乏键盘编辑功能）。

  * **评价：** 适合重度依赖语音输入且追求高质量输出的用户，"用了就回不去"。

* **豆包/火山引擎（高性价比流）：**

  * **优点：** 识别速度极快，中文及方言识别准确，免费额度大（部分赠送20小时或大量 token）。

  * **缺点：** 英文单词（如 Chrome, TablePlus）识别有时需微调配置，整合到工作流可能需要一定的开发或配置（如结合"闪电说"）。

  * **配置建议：** 推荐使用火山引擎的豆包流式语音识别模型，或直接使用豆包 1.8 模型，性价比极高。

* **微信输入法（极速免费流）：**

  * **优点：** 完全免费，响应速度极快（1-2秒），中文识别优秀，支持跨端。

  * **缺点：** 深度推理和结构化能力弱于大模型加持的工具，英文识别相对较弱。

### 2. 支付与风控：Stripe 封号潮与港卡自救

出海收款环节近期风声鹤唳，多个群友反馈 Stripe 账号被封或触发审核。

* **Stripe 预警：**

  * **高危动作：** 单量突然猛增、争议率超过 1% 极易触发风控。

  * **现状：** 即使是美国公司主体+Mercury 银行账户，也面临审核封号风险。审核期间资金可能被冻结（有群友被扣 1 万多刀需半年后处理）。

  * **策略：** 争议订单建议直接退款，不要硬刚。代码层面建议同时接入 **Creem** 作为备用，一旦 Stripe 被封立即切换。

* **Creem 配置坑点：**

  * 绑定支付宝或银行账户时，可能会遇到进度条回退或显示未绑定的 UI bug。建议耐心等待人工审核邮件，或尝试多次绑定。

* **香港开户热潮：**

  * 鉴于支付风控，不少群友计划肉身赴港开户。

  * **经验：** 提前下载好银行 APP，利用"e账户"服务，过关后连接香港网络即可快速申请。推荐汇丰、中银、工银亚洲等。

### 3. 技术避坑：Claude Code 与 OpenClaw 调优

大家在使用 Claude Code (CC) 和 OpenClaw (小龙虾) 进行开发时，总结了一套成熟的避坑与优化方案。

* **Claude Code (CC) 降本增效：**

  * **Token 刺客：** 有群友发现 CC 问个好都能消耗 100k 上下文。

  * **优化方案：**

    1. 调整 `Claude-MEM` 插件配置，减少历史记录拉取；

    2. 关闭默认的 Agent Team 模式（默认注入 4k token）；

    3. 清理插件市场中乱七八糟的 Example Skill；

    4. 关闭 `chrome-dev mcp`，改用 `agent browser` skill。

  * 经过优化，上下文占用可从 100k 降至 40k，大幅节省成本。

* **OpenClaw 部署与报错：**

  * **1008 错误：** 通常是因为 `.env` 中的 `GATEWAY_TOKEN` 与 `openclaw.json` 中的 `auth token` 不一致。

  * **Telegram 串台：** 多 Agent 共用 Bot 时会出现消息串台，建议做 Bot 隔离或通过主 Agent 分发任务。

* **账号与代理风险：**

  * **反重力（Antigravity）封号：** 学生优惠 Pro 账号用于反代（尤其是给 OpenClaw/CC 跑自动化任务）极易被封，建议仅用于轻量级网页开发或文档检索。

  * **替代方案：** 推荐使用按量付费的中转 API 服务，或者购买成品账号池服务，避免主账号被封。

### 4. 成本控制：订阅 vs 中转

针对昂贵的 AI 模型费用（如 Opus 4.5/4.6），群内形成了两派观点：

* **订阅派：** 适合高频使用者，通过 ChatGPT 订阅配置 OpenClaw 相对安全，或购买成品号。

* **中转派：** 适合间歇性爆发开发（如"写三天代码歇五天"）。按量付费的中转 API 成本更低，且部分优质中转商提供的 Opus 体验与官方无异。

1. **AI 编程要把控成本：** 主力用 MiniMax/Kimi/DeepSeek 等高性价比模型，贵价模型（Opus/Claude）仅作"最后一道防线"或复杂逻辑处理。
