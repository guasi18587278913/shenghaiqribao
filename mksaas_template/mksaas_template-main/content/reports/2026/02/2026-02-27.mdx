### 一、 深度复盘：Raphael AI 的“自传播+自动化”增长打法


刘小排详细拆解了 Raphael AI 在零付费投放的情况下，如何实现创始人旅游期间业务持续增长。核心逻辑是 **“极致产品做口碑 + AI 员工做运营”**。


**1. 自传播：把产品做成流量引擎**

* **降低门槛：** 坚持免费、免登录、无限制，取消人机验证（大幅提升转化），全站追求极速加载（PageSpeed 90+）。
* **全球化策略：** 32国语言本地化，不仅是翻译，而是重写文案以适应当地文化。
* **极简传播：** 域名短（raphael.app），生成结果可一键分享至 Twitter/Discord/小红书，利用 V2EX/Reddit 等社区形成二次传播。

**2. OpenClaw：全自动化的“数字员工”**
OpenClaw 作为本地部署的 AI 执行引擎，承担了以下自动化任务：

* **自动增长：** 实时抓取热点生成 Prompt，根据用户分布自动优化多语言 SEO 和文案。
* **自动闭环：** 抓取用户评论/Issue，自动生成优化点并每日迭代版本。
* **自动分发：** 生成优质图片后，自动分发到各大社交平台引流。
* **自动运维：** 监控 GPU/API 状态，自动重启节点，拦截恶意请求（替代人机验证），并通过 WhatsApp/Telegram 接收创始人指令。

### 二、 技术焦点：OpenClaw 部署与 Antigravity 风波


**1. Antigravity 账号“死而复生”**

* **现象：** 前几天 Antigravity 出现大批量封号（尤其是 Pro/Ultra 账号），但今日多位群友反馈账号已突然“复活”恢复使用。
* **原因推测：** 只要反代给 OpenClaw 使用就有高封号风险。虽然官方插件或 `cliproxyapi` 方案相对好一点，但本质上反代都有风险。
* **现状：** 学生包账号复活率较低，老账号相对安全。

**2. OpenClaw 最佳部署方案讨论**
关于是部署在本地 Mac 还是云端 VPS，群内产生详细对比：

* **VPS 方案（推荐）：** 成本低（如 AWS Free Tier），使用 Ubuntu + Docker 部署。适合 24 小时在线的“大脑”（Gateway）。
* **Mac Mini 方案：** 适合作为本地“肢体”（Node），拥有完整本地权限，方便随时人工介入操作，但成本较高。
* **虚拟机方案：** 有群友尝试在 Mac 中跑 Ubuntu 虚拟机再跑 Docker，被认为架构过重，不如直接 Docker 方便。
* **注意事项：** 本地部署意味着 OpenClaw 拥有电脑完整权限，需注意隐私安全。

**3. 模型选择共识**

* **主力编程：** 首选 **Codex**，稳且支持 OAuth。
* **复杂任务：** 使用 **Opus**（虽然贵但逻辑强），适合处理文档 Review 和复杂架构。
* **日常低成本：** 推荐 **Grok Fast 4.1** 处理简单日常任务，解决不了再切 Opus。
* **避坑：** 部分中转服务商提供的模型可能混入了 Minimax 或 GLM 等国产蒸馏模型，需警惕“掺水”。

### 三、 避坑指南与工具推荐


**1. 飞书机器人 API 消耗过快**

* **问题：** 有群友反馈飞书机器人一天消耗 90% API 配额。
* **原因：** 机器人的“心跳机制”在持续消耗资源。
* **解决：** 关闭心跳，改用 Webhook 推送或 Cron 定时任务触发。

**2. React 开发报错死循环**

* **案例：** 遇到 `Maximum update depth exceeded` 报错（Ref 回调触发 setState 导致死循环）。
* **建议：** 此类 React 深度 Hook 问题，国产模型（如 GLM）容易陷入车轱辘话，建议直接使用 **Codex** 或 **Opus** 进行修复。

**3. API Key 管理难题**

* 随着工具增多，API Key 管理变得混乱。群友推荐方案：
* **CC Switch：** 专门用于切换 Claude Code 的 API。
* **CLIProxyAPI：** 代理工具。
* **通用方案：** 使用 1Password、Bitwarden 或 Enpass 等密码管理软件统一存储。

1. **运营启示：** 学习 Raphael AI 的思路，不仅用 AI 写代码，更要用 AI (OpenClaw) 做 24 小时自动化的运营和分发。
